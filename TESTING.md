# SignLock 手测矩阵

这份文档用于发布前手工验收。建议至少准备 2 个普通玩家账号和 1 个管理员账号。

## 测试环境

- 服务端：Spigot、Paper 或 Folia
- Java：与当前构建环境一致
- 插件命令：`/signlock` 或别名 `/bl`
- 测试方块：单箱、双箱、木桶、门、熔炉

## 核心流程

1. 玩家手持普通牌子右键单箱，确认自动生成锁牌并消耗 1 个牌子。
2. 玩家手持普通牌子右键双箱任意半边，确认双箱整体被同一把锁保护。
3. 玩家对已上锁容器使用 `Shift + 右键`，确认不会错误创建新锁。
4. 所有者右键自己的锁牌，确认可以编辑授权内容且结构行不会被破坏。
5. 非所有者右键或编辑锁牌，确认会被阻止并收到提示。

## 命令与授权

1. 对准有效锁牌执行 `/bl add <玩家名>`，确认玩家被加入授权列表。
2. 主锁牌写满后继续执行 `/bl add <玩家名>`，确认会自动生成 `[更多用户]` 扩展牌。
3. 对准锁牌执行 `/bl remove <玩家名>`，确认该玩家被移除。
4. 移除最后一个扩展牌中的最后一名玩家后，确认空扩展牌会被清理。
5. 对准锁牌执行 `/bl info`，确认显示所有者、授权玩家和扩展牌数量。
6. 执行 `/signlock reload`，确认配置重载成功并返回正确中文提示。

## 保护矩阵

1. 未授权玩家尝试打开单箱、双箱、门、熔炉，确认会被阻止。
2. 未授权玩家尝试破坏受保护方块本体，确认会被阻止。
3. 未授权玩家尝试破坏主锁牌或扩展牌，确认会被阻止。
4. 漏斗输入或抽取已上锁容器，确认会被阻止。
5. 活塞推动锁牌、受保护方块，或推动方块撞向受保护结构，确认会被阻止。
6. 水流或爆炸影响锁牌和受保护方块，确认会被阻止。

## 身份与兼容性

1. 被授权玩家改名后重新进服，确认旧锁仍可正常访问。
2. 管理员在开启 `admin-bypass` 时验证可绕过管理限制。
3. Folia 环境下多玩家同时加锁、授权、开箱，确认没有报错。

## 发布前结论

- 所有核心流程通过：可以发布
- 若任一步失败：先修复，再重新执行受影响条目
